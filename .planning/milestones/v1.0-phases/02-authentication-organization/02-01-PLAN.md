**IMPORTANT NOTE on token operations:** The registration flow and password reset flow create tokens BEFORE a tenant context exists (user isn't authenticated yet). These operations should use the `db` client directly (not `withTenantContext`), connecting as the main database user. The RLS policies above are for when authenticated users interact with their own tokens. For token creation during registration, the operation runs outside RLS context (the connection string uses `app_user` but there's no `SET LOCAL` call, so the RLS policy check for `current_setting('app.current_tenant_id')` would fail).

**Resolution:** Token creation/verification in auth actions should use the `db` directly without `withTenantContext()`. The auth actions (register, verify, reset) all operate outside tenant context. Consider if the `email_verification_token` and `password_reset_token` tables should NOT have RLS enabled, since they're only used in unauthenticated flows. The safer approach: do NOT enable RLS on these two token tables (they don't contain tenant data and are accessed in unauthenticated context). Just grant permissions to app_user.

Revised: Only enable RLS on `account` and `auth_session`. Grant simple access on `verification_token`, `email_verification_token`, and `password_reset_token` without RLS.
