1. Create `src/lib/validations/action-item.ts` with Zod schemas for standalone action item operations:
   - `standaloneUpdateActionItemSchema`: accepts id (uuid), and optional fields: title (1-500 chars), description (string or null), assigneeId (uuid), dueDate (string or null), status (enum: "open", "completed" -- only two states per user decision, map "completed" to "completed" in DB)
   - Export the inferred types

2. Create `src/app/api/action-items/route.ts` with GET handler:
   - Auth check (401 if not authenticated)
   - Uses withTenantContext to query all open action items (status "open" or "in_progress") across all series the user participates in (as manager or report) or all series if admin
   - Query pattern: actionItems INNER JOIN sessions INNER JOIN meetingSeries INNER JOIN users (assignee) -- single query, no N+1
   - Select fields: id, title, description, status, dueDate, category, assigneeId, createdAt, sessionId, plus from JOINs: assignee firstName/lastName, seriesId, sessionNumber, reportId, report firstName/lastName (for group headers)
   - Also join the report user for series group header display
   - Filter: tenantId matches, status is "open" or "in_progress", user is participant (managerId or reportId) or admin
   - Order by: dueDate ASC NULLS LAST (overdue first, then upcoming, then no date)
   - Return: `{ actionItems: [...] }` with all fields needed for the UI

3. Create `src/app/api/action-items/[id]/route.ts` with PATCH handler:
   - Auth check (401)
   - Parse body with standaloneUpdateActionItemSchema
   - Authorization chain inside withTenantContext: fetch action item -> join session -> join series -> verify user is managerId or reportId (or admin)
   - If not found or not authorized, return 404/403
   - Build update set from provided fields (only update fields that are present in the body)
   - If status changes to "completed", set completedAt to now(); if status changes to "open", clear completedAt
   - Update updatedAt to now()
   - Return updated action item
