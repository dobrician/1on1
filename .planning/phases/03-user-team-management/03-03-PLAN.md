**Step 1: Create GET /api/users/route.ts** (list users):
- Auth check: session required
- Role check: ALL roles can see the user list (needed for manager selectors etc), but the data returned varies:
  - Admins: see all users in tenant
  - Managers: see all users (needed for team/manager views)
  - Members: see all users (needed for org directory, but limited fields)
- Query: `withTenantContext` -> select all users where tenantId matches, include team memberships via a left join or separate query
- Also include pending invites: query invite_tokens where acceptedAt is null and expiresAt > now, merge into results with status="pending"
- Return shape: array of `{ id, firstName, lastName, email, role, jobTitle, avatarUrl, managerId, managerName, isActive, status ("active" | "pending" | "deactivated"), teams: [{id, name}], invitedAt, createdAt }`
- Sort by: lastName, firstName by default
- Support query params: ?search=, ?role=, ?status= for server-side filtering (but client-side filtering is also fine for v1 -- pick one approach and be consistent)
- Decision: Use client-side filtering for v1 (Server Component fetches all users, TanStack Table handles the rest). The GET endpoint returns all users without pagination.

**Step 2: Create GET/PATCH/DELETE /api/users/[id]/route.ts**:

**GET (single user):**
- Auth check required
- Fetch user by ID within tenant context
- Include team memberships, manager info
- All roles can view (needed for profile views)
- Return 404 if not found

**PATCH (update user):**
- Auth check required
- Determine what's being updated by inspecting the body:
  - If `role` field present: admin-only. Parse with updateUserRoleSchema. CRITICAL: Check if target user is the last admin -- if so and new role is not admin, return 400 "Cannot remove the last admin". Log "role_changed" with { previousRole, newRole }.
  - If `managerId` field present: admin-only. Parse with assignManagerSchema. CRITICAL: Prevent circular references -- walk up the manager chain (max 10 levels) to verify the target managerId does not eventually point back to the user being edited. Log "manager_assigned" with { previousManagerId, newManagerId }.
  - If profile fields (firstName, lastName, jobTitle, avatarUrl): self OR admin. Parse with updateProfileSchema. If not self and not admin, return 403. Log "profile_updated".
- All updates go through withTenantContext
- Set updatedAt to now() on every PATCH
- Return updated user object

**DELETE (deactivate user):**
- Auth check + requireRole "admin"
- Cannot deactivate yourself (return 400)
- Cannot deactivate the last admin (count admins, if this is the last one, return 400)
- Set isActive = false (not a real delete -- soft delete per USER-06)
- Log "user_deactivated" audit event
- Return 200 with { success: true }

**Reactivation:** Also support PATCH with `{ isActive: true }` for reactivation (admin only). Log "user_reactivated".
