    2. **Export getTransport and getEmailFrom from send.ts**: Make `getTransport()` and `getEmailFrom()` exported (currently module-private). This is the single source of truth for SMTP transport.

    3. **Consolidate duplicate transports**: Update `src/app/api/invites/route.ts` and `src/app/api/invites/resend/route.ts` to import `getTransport` and `getEmailFrom` from `@/lib/email/send` instead of duplicating the transport creation. Remove the `"use server"` directive from send.ts if it prevents import from API routes (API routes are server-only already) -- or keep it if it doesn't cause issues.

    4. **Create `src/lib/notifications/queries.ts`**: DB query helpers:
       - `getPendingNotifications(limit?: number)`: SELECT from notifications WHERE status='pending' AND scheduledFor <= NOW(), ordered by scheduledFor ASC, with limit (default 50). Use adminDb (cross-tenant).
       - `markNotificationSent(id: string)`: UPDATE status='sent', sentAt=NOW()
       - `markNotificationFailed(id: string, error: string)`: UPDATE status='failed', error=error
       - `cancelSeriesNotifications(seriesId: string, types?: string[])`: UPDATE status='cancelled' WHERE referenceId=seriesId AND referenceType='series' AND status='pending' AND type IN (types)

    5. **Create `src/lib/notifications/scheduler.ts`**: Functions to schedule notifications:
       - `scheduleSeriesNotifications(params: { tenantId, seriesId, managerId, reportId, nextSessionAt, reminderHoursBefore, managerName, reportName })`:
         a. Cancel existing pending notifications for this series (pre_meeting + agenda_prep types)
         b. Compute reminderAt = nextSessionAt - reminderHoursBefore * 3600000
         c. Compute agendaPrepAt = nextSessionAt - 48 * 3600000
         d. Insert 4 notification records (2 for pre_meeting: manager + report, 2 for agenda_prep: manager + report), each with: tenantId, userId, type, channel='email', referenceType='series', referenceId=seriesId, scheduledFor, status='pending'
         e. Skip creating a notification if its scheduledFor is in the past (e.g., next session is less than 48h away, agenda prep would be in the past)
       - Uses adminDb (creates cross-tenant records for cron to process)

    6. **Create `src/lib/notifications/sender.ts`**: Process a single notification:
       - `processNotification(notification)`: Switch on notification.type:
         - `pre_meeting`: Fetch series + users, render PreMeetingReminderEmail, send via getTransport().sendMail()
         - `agenda_prep`: Fetch series + users + AI nudges (for manager), render AgendaPrepEmail with correct variant, send
         - `session_summary`: (stub for now -- wired in Plan 02) Log a warning that session_summary should be event-driven, not cron-scheduled
       - Each case: look up required data using adminDb with notification's tenantId/userId/referenceId, render the template via `render()` from @react-email/render, send via getTransport().sendMail()
       - For agenda_prep manager variant: fetch non-dismissed nudges from ai_nudges table for the series

    7. **Create `src/app/api/cron/notifications/route.ts`**: GET endpoint:
       - Auth: Check `request.headers.get("authorization") === Bearer ${process.env.CRON_SECRET}`. Return 401 if invalid.
       - Use the optimistic claim pattern to prevent double-sends: UPDATE notifications SET sentAt=NOW() WHERE status='pending' AND scheduledFor<=NOW() RETURNING * (atomic claim). Then process each claimed notification. On success: set status='sent'. On failure: set status='failed', error=message, sentAt=null.
       - Return JSON with processed/sent/failed counts.
       - No retry logic in v1 -- failed notifications stay failed (per research recommendation).
