5. Update `src/i18n/request.ts` to import and spread the new `validation.json` namespace alongside existing namespaces.

6. Create `src/lib/i18n/zod-error-map.ts` exporting `useZodI18nErrors()` hook:
   - Import `useTranslations` from 'next-intl' and `z, ZodIssueCode` from 'zod'
   - Call `useTranslations('validation')` to get `t`
   - Call `z.setErrorMap()` with a function that maps:
     - `too_small` with string type + minimum=1 -> `t('required')`
     - `too_small` with string type -> `t('minLength', { min: issue.minimum })`
     - `too_big` with string type -> `t('maxLength', { max: issue.maximum })`
     - `invalid_string` with email validation -> `t('invalidEmail')`
     - `invalid_type` with received='undefined' -> `t('required')`
     - Default -> `ctx.defaultError`
   - The hook returns void, just sets the global error map with current locale translations

7. Create `src/lib/i18n/api-error-toast.ts` exporting `useApiErrorToast()` hook:
   - Import `useTranslations` from 'next-intl' and `toast` from 'sonner'
   - Call `useTranslations('common')` to get `t`
   - Export a function `showApiError(error: unknown)` that:
     - If error is an Error with message matching known patterns (contains "permission", "unauthorized", "forbidden", "not found", "rate limit"), map to the corresponding `t('errors.*')` key
     - If error is an Error with message starting with "Failed to ", use `t('errors.failedTo', { action: extractedAction })`
     - For generic/unknown errors, use `t('errors.serverError')`
     - Call `toast.error()` with the translated message
   - The hook returns `{ showApiError }` for use in catch blocks

IMPORTANT: The hooks use `useTranslations` which requires React context -- they must be called inside a component, not in a utility function. Keep schemas in `src/lib/validations/` language-agnostic (no t() calls in schemas).
