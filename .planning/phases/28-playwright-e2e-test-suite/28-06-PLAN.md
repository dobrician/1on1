  <action>
Read `src/lib/db/seed.ts` first.

All ANSWER_* UUID constants follow the pattern `'66666666-XXXX-4000-6000-XXXXXXXXXXXX'`. The `6000` group is the `clock_seq_hi_res` field — RFC4122 requires this to start with bits `10` (i.e., hex digit 8, 9, a, or b). `6` starts with bits `0110` which is invalid.

**The fix:** Replace `6000` with `8000` in ALL answer UUID constant definitions (the string values, not usage sites — the usage sites reference the constants by name and will automatically get the corrected values).

**Scope:**
- 45 constants total, all following: `'66666666-XXXX-4000-6000-XXXXXXXXXXXX'`
- Constants are named: ANSWER_EVE_S1_*, ANSWER_EVE_S2_*, ANSWER_FRANK_S1_*, ANSWER_FRANK_S2_*, ANSWER_GRACE_S1_*, ANSWER_GRACE_S2_*, ANSWER_S1_*, ANSWER_S2_*
- Do NOT change any SESSION_* or other non-ANSWER constants
- Do NOT change the first group (`66666666`) — only the `6000` group

**Verification pattern for the edit:**
Before: `'66666666-0001-4000-6000-000000000001'`
After:  `'66666666-0001-4000-8000-000000000001'`

After editing, verify: `grep -c "6000" src/lib/db/seed.ts` should return 0 (no remaining 6000 variant bits in UUID strings). Note: if `6000` appears elsewhere in a non-UUID context, verify those are not answer UUID strings.
